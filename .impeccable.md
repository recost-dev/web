# Recost — Design Context

## Design Context

### Users
Indie developers and startup engineering teams who have instrumented their projects with the Recost SDK. They arrive at the dashboard already invested — they've written code to get here. The primary job: understand exactly where API spend is going across providers and act on it. Secondary job: prove the cost picture to a team lead or co-founder. These are people comfortable in terminal and code editors — they expect density, precision, and zero hand-holding.

### Brand Personality
**Techy. Sleek. Precise.**

The emotional target is "informed command" — not anxious urgency, not gamified delight. The user should feel like they're looking at an instrument panel, not a marketing site. Every element exists because it surfaces information. Nothing decorates.

References:
- **Stripe** (dark mode): data clarity, typography hierarchy, polish without flash
- **Linear**: tight spacing, monochrome + single accent, every pixel intentional

Anti-references:
- No purple gradients, no Inter, no card-soup layouts
- No Vercel-template aesthetics dropped in without thought
- No pastel/rounded playful SaaS (Notion clones, soft corners for friendliness)
- No generic AI-tool aesthetics (cyan-on-dark, glowing accents)

### Aesthetic Direction
- **Theme**: Dark only. `#0f0f0f` base, not pure black.
- **Accent**: Amber/gold — away from the inherited eco-green. Cost intelligence has money connotations; a burnished amber (`oklch(72% 0.16 65)` ≈ `#c8861a`) reads as value, precision, and financial relevance. Used sparingly — only where it carries signal.
- **Typography**: Refined grotesque for UI, monospace only for actual data/code — not as decoration. Avoid Inter/Geist as primary; lean into something with more character.
- **Spacing**: Stripe-school — tight where data is dense, generous where hierarchy needs breathing room. No uniform padding everywhere.
- **Shape**: Sharp edges. No rounded-full. Minimal border-radius (`4px` max for inputs, `6px` for cards if cards are used at all).
- **Motion**: State changes only. No decorative entrances on synchronous data. Animate transitions, not presence.

### Design Principles
1. **Data first** — every element exists to surface information. If it doesn't carry meaning, remove it.
2. **Earned density** — pack information tightly where it matters; give breathing room only where hierarchy demands it.
3. **Accent as signal** — amber appears only where it means something (active state, metric highlight, CTA). Never decorative.
4. **Precision over personality** — sharp edges, tight type, no softening for approachability. The interface is a tool, not a friend.
5. **Motion earns its place** — animate state changes and feedback only. Never animate something that was already rendered on mount.
